# Phase 05.4: Live monitoring of device state - Context

**Gathered:** 2026-03-17
**Status:** Ready for planning
**Source:** User requirements (conversation)

<domain>
## Phase Boundary

This phase delivers three features on the plugin's web UI:
1. Live monitoring — real-time device state updates on the status page
2. Blinds tab — a new tab showing blind/shutter devices with specific columns
3. Link Quality column — add LQI to the existing Device Status tab

</domain>

<decisions>
## Implementation Decisions

### Live Monitoring
- Device state should update in real-time (or near-real-time) without requiring a manual page refresh
- State data comes from Z2M's state.json which is already read server-side

### Blinds Tab
- New tab alongside existing tabs on the status page
- Shows only devices with friendly_name starting with "MS-108ZR"
- Columns: Position, State, Motor Reversal (from Z2M state data for each device)
- Data source is the same Z2M state.json already loaded

### Link Quality Column
- Add a "Link Quality" (LQI) column to the existing Device Status tab
- Value comes from the device's Z2M state data (linkquality field)

### Claude's Discretion
- Live monitoring implementation approach (polling interval, AJAX vs SSE vs WebSocket)
- Blinds tab UI layout and styling (consistent with existing tabs)
- How to handle missing state data for blind devices
- Link Quality column position in the table
- Whether Link Quality should be sortable

</decisions>

<specifics>
## Specific Ideas

- Z2M state.json is already loaded server-side in PHP (lines 367-370 of index.php)
- The device status page uses jQuery Mobile tabs
- Device state is available as data attributes on badge spans (data-z2m-state, added in Phase 5.3)
- MS-108ZR is a specific blind/shutter device model
- linkquality is a standard Z2M state field present on most devices

</specifics>

<deferred>
## Deferred Ideas

None — all items are in scope for this phase.

</deferred>
