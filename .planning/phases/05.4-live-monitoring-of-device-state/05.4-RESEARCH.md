# Phase 05.4: Live Monitoring of Device State - Research

**Researched:** 2026-03-17
**Domain:** PHP/jQuery Mobile web UI with AJAX polling, Z2M device state rendering
**Confidence:** HIGH

## Summary

This phase adds three features to the existing status page: (1) live monitoring via AJAX polling so device state updates without page refresh, (2) a new Blinds tab filtering MS-108ZR devices with Position/State/Motor Reversal columns, and (3) a Link Quality (LQI) column on the existing Device Status table.

The codebase already has all the data plumbing in place. Z2M state.json is read server-side (index.php lines 367-370) and is already available per-device via `$z2m_state_data[$ieee]`. The Device Status table already uses jQuery, AJAX (for exclusion toggles), and `data-sort-value` attributes for sortable columns. The UI uses jQuery Mobile tabs (`data-role="tabs"`) and inline JS with event delegation patterns established in Phase 5.2/5.3.

**Primary recommendation:** Use simple AJAX polling (setInterval + jQuery.getJSON) against a new PHP endpoint that returns device data as JSON. This keeps the architecture consistent with existing patterns (exclusion AJAX handler) and avoids introducing WebSockets or SSE infrastructure for a cron-driven data source that only changes every few minutes.

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| AJAX polling | Server-Sent Events (SSE) | SSE keeps connection open; overkill since data only changes on cron run (every 5-60 min); adds complexity; PHP SSE needs output buffering tricks |
| AJAX polling | WebSocket | Requires a persistent daemon (explicitly out of scope per REQUIREMENTS.md); massive over-engineering for cron-driven data |
| jQuery.getJSON | fetch API | jQuery already loaded; fetch needs polyfill for older browsers Loxberry may run on |

**Recommendation: AJAX polling at 30-second interval.** The underlying data only changes when the cron job runs (minimum every 5 minutes). A 30s poll is responsive enough while being lightweight. Uses existing jQuery patterns.

## Architecture Patterns

### Recommended Approach

```
index.php (existing)
  |
  +-- POST action=get_status_data  --> returns JSON: { devices: [...], z2m_state: {...}, last_run: "..." }
  |
  +-- <div id="tab-status">         --> existing Device Status tab (add LQI column)
  |
  +-- <div id="tab-blinds">         --> NEW Blinds tab (filtered MS-108ZR devices)
  |
  +-- <script> setInterval poll     --> AJAX replaces table bodies every 30s
```

### Pattern 1: AJAX Polling Endpoint
**What:** Add a new POST action handler in index.php that returns device + Z2M state data as JSON.
**When to use:** When the JS polling timer fires.
**Example:**
```php
// In the POST handler section of index.php
if ($_POST['action'] === 'get_status_data') {
    header('Content-Type: application/json');
    // Re-read state.json and z2m state.json (fresh data)
    $state_raw = @file_get_contents(LBPDATADIR . '/state.json');
    $state = $state_raw ? json_decode($state_raw, true) : array();
    $devices = isset($state['devices']) ? $state['devices'] : array();

    // Re-read z2m state for live values
    $z2m_state_raw = @file_get_contents($z2m_data_path . '/state.json');
    $z2m_state = $z2m_state_raw ? json_decode($z2m_state_raw, true) : array();

    echo json_encode(array(
        'devices' => $devices,
        'z2m_state' => $z2m_state,
        'last_run' => isset($state['last_run']) ? $state['last_run'] : null,
    ));
    exit;
}
```

### Pattern 3: Blinds Tab with jQuery Mobile
**What:** A new tab added to the existing `data-role="tabs"` container.
**When to use:** Shows filtered view of MS-108ZR devices.
**Key concern:** jQuery Mobile tabs require the tab panel to exist in the HTML at page load. Adding tabs dynamically after JQM initialization is fragile. The tab should be rendered server-side in PHP.

### Anti-Patterns to Avoid
- **Do NOT use WebSocket/SSE:** The data source is file-based and cron-driven. Real-time push adds complexity with zero benefit.
- **Do NOT poll faster than 15 seconds:** Z2M writes state.json on device events but reading the file too frequently adds unnecessary I/O. 30s is the sweet spot.
- **Do NOT diff individual cells:** Rebuilding tbody HTML on each poll is simpler, avoids stale DOM issues, and the table is small enough (typically <50 devices) that performance is not a concern.
- **Do NOT add the Blinds tab dynamically via JS:** jQuery Mobile tabs must exist in the DOM at page creation time. Render the tab panel in PHP.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| AJAX polling | Custom XHR wrapper | jQuery.post() or jQuery.getJSON() | jQuery already loaded, handles errors, provides promise API |
| Tab UI | Custom tab switching | jQuery Mobile `data-role="tabs"` | Already used, handles ARIA, mobile responsive |
| Table sorting | Custom sort for new columns | Existing `sortTable()` function | Already in index.php, supports 'str' and 'num' types |

## Common Pitfalls

### Pitfall 1: jQuery Mobile Tab Initialization Order
**What goes wrong:** Adding a new tab panel to the `data-role="tabs"` container after jQuery Mobile has initialized causes the tab to not render or behave correctly.
**Why it happens:** JQM processes tabs during `pagecreate` event. Late additions miss this lifecycle.
**How to avoid:** Render the Blinds tab panel in PHP HTML alongside the existing tabs. Use `<?php if (...): ?>` for conditional content, but the tab header and empty panel must exist.
**Warning signs:** Tab header appears but clicking does nothing; tab content not visible.

### Pitfall 2: Polling Continues When Tab Not Visible
**What goes wrong:** AJAX requests fire even when the user is on the Settings tab or the browser tab is inactive, wasting bandwidth and server resources.
**Why it happens:** setInterval runs unconditionally.
**How to avoid:** Check if the status/blinds tab is active before polling. Use `document.hidden` to pause when browser tab is inactive. Clear interval on Settings tab, restart on Status tab.

### Pitfall 3: Z2M State Key Mismatch
**What goes wrong:** The Blinds tab filter uses `friendly_name` prefix but Z2M state.json is keyed by IEEE address, not friendly_name.
**Why it happens:** Need to correlate IEEE keys in state.json with friendly_name from the watchdog state.json devices map.
**How to avoid:** The watchdog's state.json already stores `friendly_name` per device alongside the IEEE key. Filter devices from the watchdog state, then look up Z2M state by IEEE address.

### Pitfall 4: Missing State Fields for Blind Devices
**What goes wrong:** Not all MS-108ZR devices may have all fields populated in state.json (e.g., position is null until first movement).
**Why it happens:** Z2M only writes fields when a device reports them. A newly paired device may have minimal state.
**How to avoid:** Display "N/A" or "--" for missing fields. Never assume position/state/motor_reversal exist.

### Pitfall 5: z2m_data_path Not Available in AJAX Handler
**What goes wrong:** The AJAX JSON endpoint needs the z2m_data_path to read state.json, but this value is computed after the POST handlers in current code flow.
**Why it happens:** The auto-detect logic runs after POST handlers in the current index.php structure.
**How to avoid:** Move z2m_data_path resolution (config read + auto-detect) to run before POST handlers, or duplicate the resolution in the AJAX handler. The cleanest approach: extract path resolution to a helper function called from both locations.

### AJAX Polling Pattern (consistent with existing exclusion AJAX)
```javascript
var pollTimer = null;
var POLL_INTERVAL = 30000; // 30 seconds

function startPolling() {
    if (pollTimer) return;
    pollTimer = setInterval(function() {
        if (document.hidden) return; // Skip when tab not visible
        jQuery.post('index.php', { action: 'get_status_data' }, function(data) {
            updateStatusTable(data);
            updateBlindsTable(data);
        }, 'json').fail(function() {
            // Silent fail -- next poll will retry
        });
    }, POLL_INTERVAL);
}

function stopPolling() {
    if (pollTimer) { clearInterval(pollTimer); pollTimer = null; }
}
```

## Open Questions

1. **Polling and fresh z2m state.json data**
   - What we know: z2m writes state.json on each device event. The cron watchdog writes state.json on each run.
   - What's unclear: Whether reading z2m's state.json on every 30s poll introduces any lock contention or mid-write corruption risk.
   - Recommendation: Use the same retry-on-parse-error pattern from z2m-reader.js (try parse, on failure retry once). The file is small and writes are near-atomic on Linux.

2. **LQI value interpretation**
   - What we know: linkquality is 0-255 (standard ZCL LQI). Higher is better.
   - What's unclear: Whether to show raw number or interpret as signal strength indicator.
   - Recommendation: Show raw number for v1. A color-coded signal indicator could be a future enhancement.
