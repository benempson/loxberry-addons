# Phase 5.1: Replace MQTT with Direct Z2M File Reading - Context

**Gathered:** 2026-03-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the MQTT broker dependency with direct reading of zigbee2mqtt's local data files (state.json, database.db). Eliminates MQTT connection, MQTT configuration in the UI, and the mqtt npm dependency. Updates the web UI to show a z2m data path field instead of MQTT settings. Rewrites bridge offline detection to use systemctl + file freshness checks. Clean break — no backward compatibility with old MQTT config (pre-release 0.1.0, no external users).

</domain>

<decisions>
## Implementation Decisions

### Z2M Data Sources
- Read device state (last_seen, battery) from z2m's `state.json`
- Read device registry (IEEE, friendly_name, power_source, type) from z2m's `database.db`
- Z2M data path: `/opt/zigbee2mqtt/data` on the user's system (non-standard — not a Loxberry plugin install)
- Auto-detect z2m data path at runtime: search known paths (`/opt/zigbee2mqtt/data`, `/opt/loxberry/data/plugins/zigbee2mqtt/zigbee2mqtt/`, etc.)
- Auto-detected path shown in UI and overridable by user
- If z2m_data_path is empty in config, auto-detect; if set, use the configured path

### Config UI Changes
- Remove entire MQTT settings section (host, port, username, password, base topic)
- Replace with single "Z2M Data Path" field showing auto-detected or configured path
- Status line below field shows: device count, state.json age (e.g., "67 devices, state.json 2m ago")
- "Verify Path" button replaces "Test MQTT Connection" — checks path exists, state.json readable, database.db readable, shows device count and freshness
- Remove "Test MQTT Connection" button entirely
- Keep "Send Test Email" button (still relevant)

### Bridge Offline Detection
- Primary check: `systemctl status zigbee2mqtt` — process running or not
- Secondary check: state.json file modification time freshness — catches running-but-frozen z2m
- Alert if either check fails
- Replaces the MQTT bridge/state topic approach entirely

### Backward Compatibility
- Clean break — no migration from old MQTT config
- Old MQTT fields in watchdog.cfg are ignored (replaced with Z2M section)
- Existing state.json device data preserved (last_seen, battery, alerts all still valid)
- postinstall.sh leaves z2m_data_path empty — auto-detection happens at runtime
- No migration code needed (pre-release 0.1.0, only user is the developer)

### Modules to Remove/Replace
- Remove `bin/lib/mqtt-collector.js` — replaced with z2m file reader
- Remove `bin/test-mqtt.js` — replaced with verify-z2m-path logic
- Remove `mqtt` npm dependency from package.json and bin/package.json
- Rewrite `bin/lib/device-registry.js` — read from database.db instead of bridge/devices MQTT message
- Rewrite `bin/lib/bridge-monitor.js` — use systemctl + file freshness instead of MQTT bridge/state
- Update `bin/watchdog.js` — no MQTT connection, read z2m files directly
- Update `bin/lib/config.js` — replace MQTT section with Z2M section in defaults
- Update `webfrontend/htmlauth/index.php` — replace MQTT form fields with z2m path field
- Update `postinstall.sh` — default config changes (no MQTT, add Z2M section)

### Claude's Discretion
- Exact auto-detect path search order and logic
- database.db parsing approach (it's a JSON file despite the .db extension)
- state.json freshness threshold for secondary bridge check
- How to handle race conditions reading z2m's state.json while z2m is writing it
- INI section naming for the new Z2M config
- Test structure for new modules

</decisions>

<specifics>
## Specific Ideas

- Z2M's database.db is actually a JSON file (not SQLite) containing device metadata including IEEE address, friendly_name, power_source, type
- Z2M's state.json maps friendly_name → { last_seen, battery, ... } — need to cross-reference with database.db for IEEE addresses
- The drain_seconds config field becomes irrelevant and should be removed from CRON section
- The `mqtt` npm package is the largest dependency — removing it simplifies the plugin significantly
- Auto-detect should try common paths: `/opt/zigbee2mqtt/data`, `/opt/loxberry/data/plugins/zigbee2mqtt/zigbee2mqtt/`, `/opt/loxberry/data/plugins/zigbee2mqtt/`

</specifics>
