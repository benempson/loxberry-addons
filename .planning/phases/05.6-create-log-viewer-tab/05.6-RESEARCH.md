# Phase 05.6: Create Log Viewer Tab - Research

**Researched:** 2026-03-19
**Domain:** Node.js structured logging + PHP log viewer UI (jQuery Mobile)
**Confidence:** HIGH

## Summary

This phase adds structured logging to the watchdog plugin and a "Logs" tab to the web UI. The backend work involves: (1) creating a Node.js logger module that writes structured, rotated log files, (2) instrumenting all existing code paths (watchdog runs, cron, config saves, notifications, errors) to emit log entries, (3) redirecting cron output from `/dev/null` to the log system, and (4) adding a PHP AJAX endpoint that reads/filters/paginates log data. The frontend work involves adding a fourth tab with a filtered table, severity badges, auto-polling, and highlight animations for new entries.

The codebase already has well-established patterns for AJAX endpoints (Phase 5.4 `get_status_data`), table rendering with innerHTML rebuild, polling with `setInterval`, and jQuery Mobile tabs. The log viewer reuses all of these patterns with log-specific filtering logic.

**Primary recommendation:** Create a simple line-oriented log file format (one JSON object per line, aka JSONL/NDJSON) for easy tail-reading and filtering. Use Node.js `fs` for synchronous append writes (logging must not slow down the watchdog). Implement rotation in the logger itself since this is a simple embedded plugin with no external log framework dependency.

### Pattern 1: JSONL Log Format
**What:** Each log entry is a single JSON object on one line, appended to the log file.
**When to use:** All log writes from Node.js.
**Example:**
```javascript
// Each line in watchdog.log:
{"ts":"2026-03-19T10:30:00.000Z","sev":"Info","src":"watchdog","msg":"Run complete. 42 devices tracked. No changes"}
{"ts":"2026-03-19T10:30:00.100Z","sev":"Error","src":"notify","msg":"Email delivery failed: SMTP connection timeout"}
```

**Why JSONL:**
- One `JSON.parse()` per line -- no need to parse the entire file
- Easy to read last N lines (read file, split by newline, slice)
- Easy to filter by timestamp or severity in PHP
- Append-only writes are safe even if process crashes mid-write (worst case: one truncated line)

### Pattern 2: Log Rotation in logger.js
**What:** Before each write, check file size. If over max, rotate: `watchdog.log` -> `watchdog.log.1`, `watchdog.log.1` -> `watchdog.log.2`, etc. Delete oldest beyond max count.
**When to use:** Every log write (cheap -- just a `statSync` call).
**Example:**
```javascript
function rotate(logPath, maxSize, maxFiles) {
  try {
    const stat = fs.statSync(logPath);
    if (stat.size < maxSize) return;
  } catch (e) { return; } // file doesn't exist yet

  // Shift existing rotated files
  for (let i = maxFiles - 1; i >= 1; i--) {
    const from = i === 1 ? logPath : logPath + '.' + (i - 1);
    const to = logPath + '.' + i;
    try { fs.renameSync(from, to); } catch (e) { /* file may not exist */ }
  }
  // Create fresh log file
  fs.writeFileSync(logPath, '');
}
```

### Pattern 4: Log Polling (Reuse Phase 5.4)
**What:** Use same `setInterval` pattern as device status polling, but for the Logs tab. Poll only when Logs tab is active.
**When to use:** Log tab auto-refresh.
**Recommendation:** 10-second poll interval (logs change more frequently during watchdog runs than device status does).

### Pattern 5: New Entry Highlight Animation
**What:** Track the newest timestamp seen. After each poll, entries newer than the previous newest get a CSS class that triggers a fade-out animation.
**Example:**
```css

  to { background-color: transparent; }
}

```

### Anti-Patterns to Avoid
- **Reading entire log files into memory for every poll:** Read and parse only what's needed. For "Last 100 Lines" mode, read file backwards or read all lines and slice last 100.
- **Using console.log for logging in watchdog.js:** Replace with logger calls. Console output still goes to cron, but structured logs go to the log file.
- **Blocking the watchdog main loop with log writes:** Use `fs.appendFileSync` -- synchronous but fast for small writes. Async would complicate error handling for no real benefit.

## Common Pitfalls

### Pitfall 1: Cron Redirect vs Logger
**What goes wrong:** Changing cron from `> /dev/null 2>&1` to a file redirect captures console output, but that's unstructured text, not the JSONL format.
**Why it happens:** Confusion between cron stdout redirect and the structured logger.
**How to avoid:** Keep cron redirecting to `/dev/null`. The logger writes directly to the log file via `fs.appendFileSync`. Console output remains for debug/error visibility in manual runs only.
**Warning signs:** Log file contains unstructured text mixed with JSON lines.

### Pitfall 2: Log File Permissions on Loxberry
**What goes wrong:** PHP (running as www-data or loxberry) cannot read log files created by Node.js (running as loxberry via cron).
**Why it happens:** Different users or umask settings.
**How to avoid:** Log files go in `LBPDATADIR` (`/opt/loxberry/data/plugins/zigbee_watchdog/`) which is already readable by both PHP and Node.js. Ensure `0644` permissions on log files.
**Warning signs:** PHP endpoint returns empty data despite logs being written.

### Pitfall 3: Large "All Logs" Response
**What goes wrong:** With 5 x 1MB log files, "All Logs" could return thousands of entries, making the browser slow.
**Why it happens:** No server-side pagination.
**How to avoid:** For "All Logs", still apply a reasonable server-side cap (e.g., 5000 entries). The UI won't meaningfully display more than this. Add a note like "Showing most recent 5000 entries" if truncated.
**Warning signs:** Browser tab becomes unresponsive when selecting "All Logs".

### Pitfall 4: Config Section Name Collision
**What goes wrong:** Adding a `[LOGGING]` section to watchdog.cfg could break existing config reading if not added to both PHP `$defaults` and Node.js `DEFAULTS`.
**Why it happens:** Config is read by both PHP and Node.js with explicit section/key defaults.
**How to avoid:** Add `LOGGING` section to both `$defaults` in index.php and `DEFAULTS` in config.js simultaneously. Use the same key names.
**Warning signs:** Default values not applied; config save loses logging settings.

### Pitfall 5: Tab Index Shift
**What goes wrong:** Adding a 4th tab (Logs at index 3) shifts the polling skip logic that currently checks `activeTab === 0` for Settings.
**Why it happens:** The polling code skips when Settings tab is active. Adding a tab changes indices.
**How to avoid:** Update polling logic to check tab index properly. Logs tab should trigger its own separate polling (log data, not device data). Device status polling should skip on both Settings AND Logs tabs.
**Warning signs:** Device polling runs on Logs tab (wasteful) or log polling runs on device tabs.

### Pitfall 6: PHP Log Reading Performance
**What goes wrong:** Using `file_get_contents` on all 5 rotated files then `explode("\n")` on each creates a large array in memory.
**Why it happens:** Naive "read everything then filter" approach.
**How to avoid:** For line-count modes (100/200/300 Lines), read the main log file first, only read rotated files if more lines are needed. Read in reverse order (newest first). For time-based filters, must scan all files but can stop early when entries are older than the cutoff.
**Warning signs:** PHP memory warnings; slow AJAX responses.

## Code Examples

### Logger Module (bin/lib/logger.js)
```javascript
'use strict';
const fs = require('fs');
const path = require('path');

const SEVERITIES = ['Debug', 'Info', 'Warning', 'Error'];
const DEFAULT_MAX_SIZE = 1048576; // 1MB
const DEFAULT_MAX_FILES = 5;

let logPath = null;
let maxSize = DEFAULT_MAX_SIZE;
let maxFiles = DEFAULT_MAX_FILES;

function init(filePath, opts = {}) {
  logPath = filePath;
  if (opts.maxSize) maxSize = opts.maxSize;
  if (opts.maxFiles) maxFiles = opts.maxFiles;
}

function rotate() {
  if (!logPath) return;
  try {
    const stat = fs.statSync(logPath);
    if (stat.size < maxSize) return;
  } catch (e) { return; }

  for (let i = maxFiles - 1; i >= 1; i--) {
    const from = i === 1 ? logPath : logPath + '.' + (i - 1);
    const to = logPath + '.' + i;
    try { fs.renameSync(from, to); } catch (e) {}
  }
  fs.writeFileSync(logPath, '');
}

function log(severity, source, message) {
  if (!logPath) return;
  if (!SEVERITIES.includes(severity)) severity = 'Info';
  rotate();
  const entry = JSON.stringify({
    ts: new Date().toISOString(),
    sev: severity,
    src: source,
    msg: message,
  }) + '\n';
  fs.appendFileSync(logPath, entry, 'utf8');
}

module.exports = { init, log, SEVERITIES };
```

### PHP Log Reader Endpoint
```php
if ($_POST['action'] === 'get_log_data') {
    header('Content-Type: application/json');
    $log_base = LBPDATADIR . '/watchdog.log';
    $volume = $_POST['volume'] ?? '100';
    $severity = strtolower($_POST['severity'] ?? 'all');
    $search = $_POST['search'] ?? '';

    // Collect all log entries from all files (newest file first)
    $all_entries = [];
    $files = [$log_base];
    for ($i = 1; $i <= 10; $i++) {
        if (file_exists($log_base . '.' . $i)) $files[] = $log_base . '.' . $i;
    }
    foreach ($files as $f) {
        if (!file_exists($f)) continue;
        $lines = file($f, FILE_IGNORE_NEW_LINES | FILE_SKIP_EMPTY_LINES);
        if (!$lines) continue;
        foreach ($lines as $line) {
            $entry = @json_decode($line, true);
            if (!$entry || !isset($entry['ts'])) continue;
            $all_entries[] = $entry;
        }
    }

    // Sort newest first
    usort($all_entries, function($a, $b) {
        return strcmp($b['ts'], $a['ts']);
    });

    // Apply severity filter
    if ($severity !== 'all') {
        $all_entries = array_values(array_filter($all_entries, function($e) use ($severity) {
            return strtolower($e['sev']) === $severity;
        }));
    }

    // Apply text search filter
    if ($search !== '') {
        $search_lower = strtolower($search);
        $all_entries = array_values(array_filter($all_entries, function($e) use ($search_lower) {
            return strpos(strtolower($e['msg']), $search_lower) !== false;
        }));
    }

    // Apply volume filter
    if (preg_match('/^(\d+)$/', $volume, $m)) {
        // Line count mode
        $all_entries = array_slice($all_entries, 0, intval($m[1]));
    } elseif (preg_match('/^(\d+)h$/', $volume, $m)) {
        // Time-based mode
        $cutoff = date('c', time() - intval($m[1]) * 3600);
        $all_entries = array_values(array_filter($all_entries, function($e) use ($cutoff) {
            return $e['ts'] >= $cutoff;
        }));
    }
    // "all" = no volume filter (but cap at 5000)
    if (count($all_entries) > 5000) {
        $all_entries = array_slice($all_entries, 0, 5000);
    }

    echo json_encode(['entries' => $all_entries, 'total' => count($all_entries)]);
    exit;
}
```

## Design Decisions (Claude's Discretion)

### Poll Interval: 10 seconds
**Rationale:** Logs are most useful during active debugging. 10s gives near-real-time feel without excessive server load. The watchdog runs on a cron schedule (minimum 5 minutes), so new log entries arrive in bursts rather than continuously.

### Log File Path: `LBPDATADIR/watchdog.log`
**Path:** `/opt/loxberry/data/plugins/zigbee_watchdog/watchdog.log`
**Rotated files:** `watchdog.log.1`, `watchdog.log.2`, etc.
**Rationale:** Same directory as `state.json`. Both PHP and Node.js already use `LBPDATADIR` / `DATA_DIR`. No new directory needed.

### Redirect Approach: Logger Module, Not Cron Redirect
**Rationale:** Redirecting cron stdout would capture console.log output as unstructured text. Instead, create `bin/lib/logger.js` that writes structured JSONL. Instrument `watchdog.js`, `notify.js`, and other modules to call `logger.log()`. Keep console output for backward compatibility (manual runs, debugging). The cron line stays as `> /dev/null 2>&1`.

### PHP Config Save Logging
**Rationale:** When the user saves settings via the Settings tab, the PHP save handler should also write a log entry. Since the logger is a Node.js module, PHP will append a JSONL line directly to the log file using `file_put_contents($logPath, $jsonLine . "\n", FILE_APPEND | LOCK_EX)`. This avoids needing a separate PHP logger class.

### No Virtual Scrolling / Pagination
**Rationale:** With a 5000-entry server-side cap, the HTML table will have at most 5000 rows. This is within browser capability for a simple table. Virtual scrolling would add significant complexity (scroll position management, dynamic row heights for expanded messages) for minimal benefit given the data cap.

### Tab-Specific Polling
**Recommendation:** The existing device/blinds polling should skip when the Logs tab is active (tab index 3). The log polling should only run when the Logs tab is active. Use the same `jQuery('[data-role="tabs"]').tabs('option', 'active')` pattern.

## Open Questions

1. **PHP direct log writes -- file locking**
   - What we know: PHP can append to the same JSONL log file using `FILE_APPEND | LOCK_EX`. Node.js uses `fs.appendFileSync`.
   - What's unclear: Whether `LOCK_EX` in PHP and synchronous append in Node.js could conflict if both write simultaneously.
   - Recommendation: This is extremely unlikely in practice (PHP writes only on config save; Node.js writes during cron runs). Even if concurrent, `appendFileSync` and `FILE_APPEND` are both atomic for writes under 4KB on Linux (PIPE_BUF guarantee for regular files via O_APPEND). No action needed.
