# Phase 05.3: Display Device State in Tooltip on Device Status Page - Research

**Researched:** 2026-03-17
**Domain:** PHP/jQuery Mobile UI -- tooltip, CSS fixes, event handling
**Confidence:** HIGH

## Summary

This phase delivers three focused UI changes to the existing device status page (`webfrontend/htmlauth/index.php`): a device state tooltip, a search placeholder alignment fix, and a clear-search-button bug fix. All three changes are confined to the single PHP file and require no backend/Node.js changes.

The Z2M `state.json` file (already read server-side in PHP for the Z2M status line) contains per-device state keyed by **friendly_name**. This raw JSON state data can be embedded as a `data-z2m-state` attribute on each table row and displayed in a tooltip/popup triggered from the Alert Status badge. jQuery Mobile provides a `popup` widget that works well for both hover (desktop) and tap (mobile) without requiring any additional libraries.

**Primary recommendation:** Use jQuery Mobile's built-in popup widget for the tooltip, embed Z2M state as HTML data attributes (JSON-encoded), fix search CSS with padding-left on the input, and fix the clear button by listening for the `search` event (fired by the browser when the clear button is clicked on `input[type=search]`).

## Architecture Patterns

### Data Flow for Tooltip
```
z2m state.json (on disk)
  --> PHP reads at page load (already done for Z2M status line)
  --> PHP maps friendly_name keys to IEEE addresses
  --> JSON-encoded state embedded as data-z2m-state attribute on <tr>
  --> JS reads data attribute, shows in jQuery Mobile popup on hover/click
```

### Key Detail: Z2M state.json Key Format
Z2M's `state.json` is keyed by **friendly_name** (e.g., `"Living Room Light": {"state": "ON", "brightness": 254}`), NOT by IEEE address. The PHP code must map from friendly_name (available in `$row['name']`) to the state.json entry.

### Pattern 1: jQuery Mobile Popup Widget
**What:** JQM provides `data-role="popup"` elements that can be opened programmatically.
**When to use:** When you need a positioned overlay that works on both desktop and mobile.
**Example:**
```php
<!-- One popup container, content swapped via JS -->
<div data-role="popup" id="state-popup" class="ui-content" data-theme="a"
     style="max-width:400px;max-height:300px;overflow:auto;">
    <pre id="state-popup-content" style="margin:0;font-size:0.8em;white-space:pre-wrap;"></pre>
</div>
```

```javascript
// Open popup on hover (desktop) / click (mobile)
jQuery(document).on('click', '.alert-badge[data-z2m-state]', function(e) {
    e.preventDefault();
    var state = JSON.parse(this.getAttribute('data-z2m-state'));
    jQuery('#state-popup-content').text(JSON.stringify(state, null, 2));
    jQuery('#state-popup').popup('open', { positionTo: this });
});
```

### Pattern 2: Search Input Clear Event
**What:** `input[type=search]` fires a `search` event when the user clicks the browser's native clear button (the X). It also fires an `input` event in most browsers, but the current code only debounces on `input` -- the clear button may not trigger the jQuery-delegated `input` handler properly.
**When to use:** Always listen for both `input` and `search` events on search inputs.
**Example:**
```javascript
// Add 'search' event listener alongside existing 'input' listener
el.addEventListener('search', function() { applyFilters(); });
```

### Anti-Patterns to Avoid
- **Generating one popup element per device row:** Creates DOM bloat. Use a single shared popup and swap content via JS.
- **Using title attribute for tooltip:** Title tooltips are unstyled, have browser-imposed delays, and don't work on mobile.
- **Large data attributes on hundreds of rows:** Z2M state per device is typically 5-20 properties (small). For hundreds of devices this is still under 50KB total. Not a concern for this use case.

## Common Pitfalls

### Pitfall 1: Z2M state.json keyed by friendly_name, not IEEE
**What goes wrong:** Looking up state by IEEE address returns nothing.
**Why it happens:** Z2M stores state keyed by friendly_name (e.g., `"Kitchen Sensor"`).
**How to avoid:** In PHP, after reading z2m state.json into `$z2m_state_data`, look up each device's state using `$row['name']` (friendly_name), not `$row['ieee']`.
**Warning signs:** Empty tooltip for all devices.

### Pitfall 2: jQuery Mobile popup must be initialized before opening
**What goes wrong:** `.popup('open')` throws "cannot call methods on popup prior to initialization."
**Why it happens:** JQM enhances elements on `pagecreate`. If the popup div is added after page init, it won't be enhanced.
**How to avoid:** Place the popup `<div>` in the static HTML (PHP output), not dynamically created by JS. Or call `jQuery('#state-popup').popup()` to initialize manually.
**Warning signs:** JS error in console on first click.

### Pitfall 3: HTML-encoding JSON in data attributes
**What goes wrong:** JSON containing quotes or special chars breaks the HTML attribute.
**Why it happens:** Unescaped `"` in JSON breaks the `data-z2m-state="..."` attribute.
**How to avoid:** Use `htmlspecialchars(json_encode($state), ENT_QUOTES)` in PHP.
**Warning signs:** Broken HTML, missing attributes, XSS vulnerability.

### Pitfall 4: Search clear button on jQuery Mobile wrapped inputs
**What goes wrong:** jQuery Mobile wraps `input[type=search]` in a `div.ui-input-search` and adds its own clear button. The native browser clear button may be hidden, or there may be TWO clear buttons.
**Why it happens:** JQM enhances search inputs automatically.
**How to avoid:** Listen for the `input` event on the actual input element, which fires when JQM's clear button is clicked (JQM sets the value to '' and triggers input). Also add a `change` event listener as fallback. Test on the actual Loxberry page.
**Warning signs:** Clear button clears text but filter doesn't update.

### Pitfall 5: Hover events on mobile
**What goes wrong:** `mouseenter`/`mouseover` don't work reliably on touch devices.
**Why it happens:** Touch devices simulate mouse events inconsistently.
**How to avoid:** Use `click` as the universal trigger. JQM popup works with click on both desktop and mobile. Desktop users can click the badge; hover is a nice-to-have but click is reliable.
**Warning signs:** Tooltip never appears on tablets/phones.

## Code Examples

### PHP: Read Z2M state and embed in table rows
```php
// Already have $z2m_data_path and $z2m_state_data from existing code (lines 367-370)
// In the table row loop, look up device state by friendly_name:
$device_z2m_state = isset($z2m_state_data[$row['name']]) ? $z2m_state_data[$row['name']] : null;
$state_json_attr = $device_z2m_state ? htmlspecialchars(json_encode($device_z2m_state), ENT_QUOTES) : '';
```

### PHP: Alert badge with data attribute
```php
<span class="alert-badge" style="..." data-z2m-state="<?php echo $state_json_attr; ?>">
    <?php echo htmlspecialchars($row['alert_status']); ?>
</span>
```

### JS: Popup open on click
```javascript
jQuery(document).on('click', '.alert-badge[data-z2m-state]', function(e) {
    e.preventDefault();
    e.stopPropagation();
    var raw = this.getAttribute('data-z2m-state');
    if (!raw) return;
    try {
        var state = JSON.parse(raw);
        jQuery('#state-popup-content').text(JSON.stringify(state, null, 2));
        jQuery('#state-popup').popup('open', { positionTo: this });
    } catch(ex) {}
});
```

### CSS: Search placeholder padding
```css
#device-search {
    padding-left: 30px; /* Clear the magnifying glass icon */
}
```

### JS: Fix clear search button
```javascript
// Add 'search' event (fires on native clear button click)
el.addEventListener('search', function() {
    if (searchTimer) clearTimeout(searchTimer);
    applyFilters();
});
```
