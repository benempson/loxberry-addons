## Human Verification Required

Three behaviors cannot be verified programmatically because they require jQuery Mobile popup enhancement and live browser interaction:

### 1. Z2M State Tooltip Popup

**Test:** Navigate to the Status tab. Ensure Z2M path is configured and `state.json` exists. Click an alert badge (the coloured status label in the Alert Status column).
**Expected:** A jQuery Mobile popup opens near the badge showing pretty-printed JSON of the device's raw Z2M state (e.g. `{ "battery": 87, "linkquality": 142, ... }`).
**Why human:** jQuery Mobile `popup('open')` requires the popup div to be enhanced by `pagecreate`, which only happens in a live browser. The static code is correct but popup activation cannot be asserted with grep.

### 2. Search Placeholder Alignment

**Test:** Navigate to the Status tab. Observe the search input at the top of the device table.
**Expected:** The "Filter devices..." placeholder text starts to the right of the magnifying glass icon — no overlap.
**Why human:** CSS visual alignment is not testable without rendering the page in a browser.

### 3. Clear Search Button Filter Reset

**Test:** Type a partial device name into the search filter (e.g. "sensor"). Verify the list is filtered. Then click the native clear (X) button inside the search input.
**Expected:** All devices reappear immediately without needing to manually delete the text or press Enter.
**Why human:** The browser-native `search` event (fired by the clear button) requires an interactive browser session to trigger. The event listener is wired in code; the behaviour is contingent on browser support for `input[type=search]` clear button events.

### 4. Graceful Degradation (No State Data)

**Test:** Click an alert badge for a device that is not present in `state.json` (or when Z2M path is unconfigured).
**Expected:** Nothing happens — no popup opens, no JS error in console.
**Why human:** Requires a live environment with a specific device configuration.
